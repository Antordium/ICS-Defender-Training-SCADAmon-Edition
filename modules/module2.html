üìä The Purdue Model (ISA-95/IEC 62264)

The Purdue Enterprise Reference Architecture (PERA), also known as the Purdue Model, is the foundational framework for understanding ICS network architecture. Developed at Purdue University in the 1990s and formalized in the ISA-95 standard (now IEC 62264), this model divides industrial networks into hierarchical levels based on function and security requirements.

‚ö†Ô∏è Safety Zone Alert

Level 0 and Level 1 together form the Safety Zone‚Äîthe most critical area of an ICS network. Compromise at these levels can cause physical damage, environmental disasters, or loss of human life. The 2017 TRITON/TRISIS attack specifically targeted Level 1 safety controllers at a Saudi petrochemical facility.

üõ°Ô∏è IDMZ Golden Rule

No direct communication should ever flow between Level 4 (Enterprise) and Level 3 (Site Operations). All traffic must pass through the IDMZ, where it can be inspected, logged, and controlled. Think of the IDMZ as a security checkpoint‚Äînothing crosses without being validated.

Network segmentation divides a large network into smaller, isolated segments to contain breaches and limit lateral movement. In ICS environments, segmentation is implemented using multiple technologies working together.

VLANs - Virtual Local Area Networks L2 SEG
üîÄ
Function: Layer 2 segmentation - separate broadcast domains
Use Cases: Separate vendors, process areas, criticality levels
Weakness: VLAN-hopping attacks, trunk port misconfigs
Best Practice: Always combine with L3 firewalls/ACLs!
Industrial Firewalls L3+ SEG
üî•
Capability: Deep packet inspection of Modbus, DNP3, OPC
Environment: Extended temp ranges, harsh industrial conditions
Failure Mode: Fail-open vs fail-closed - critical consideration!
Placement: Between Purdue levels, OT/IDMZ, vendor systems

‚û°Ô∏è
Function: Hardware-enforced one-way communication
Strength: Cannot be bypassed by software attacks - true air gap
Use Cases: Historian to IT, security logs to SIEM, cloud analytics
Limitation: TCP acknowledgments need protocol-break technology

üìê IEC 62443: Zones and Conduits

IEC 62443 (formerly ISA/IEC 62443) is the international standard for industrial automation and control systems security. It introduces the concepts of zones and conduits as a systematic approach to ICS network architecture.

üìã Zone Assessment Questions

  • What assets are in this zone?
  • What is the consequence of compromise?
  • What connectivity does this zone require?
  • Who needs access?
  • What threats are most relevant?
  • What security controls already exist?

üî• Real-World Incident: Ukraine Power Grid Attack (2015)

üé¨
Video: Ukraine Attack Case Study
[V2.4 - assets/videos/m2_v4_ukraine_attack.mp4]

On December 23, 2015, attackers compromised three Ukrainian power distribution companies, causing power outages affecting approximately 225,000 customers. This attack demonstrated the consequences of inadequate network segmentation and defense-in-depth.

‚ö° Target: Ukrainian Power Distribution

Attackers used spear-phishing emails with malicious Microsoft Word documents to compromise IT workstations. From there, they moved laterally through the corporate network, eventually finding VPN connections to the OT network. Once in the OT environment, they used legitimate HMI software to open breakers at substations, causing blackouts for ~225,000 customers.

Duration: Attackers had access for 6+ months before executing attack.

Architecture Failures

Failure What Should Have Been
‚ùå Direct VPN from IT to OT ‚úÖ IDMZ with jump servers, session logging
‚ùå Shared IT/OT credentials ‚úÖ Separate credential domains, PAM
‚ùå No MFA on remote access ‚úÖ MFA required for all remote connections
‚ùå No OT network monitoring ‚úÖ ICS-specific IDS/anomaly detection
‚ùå No application whitelisting ‚úÖ Whitelisting on HMI workstations

üí° Key Lessons Learned

Proper network architecture could have prevented or limited this attack:

  • IDMZ with jump servers would have provided a chokepoint for monitoring
  • Segmentation between substations would have limited attack scope
  • Application whitelisting would have prevented custom malware execution
  • MFA would have stopped credential reuse from IT to OT

Infographic: Architecture Implementation Checklist
[I2.4 - assets/images/m2_i4_implementation_checklist.png]

Building a New ICS Network

  1. Start with the Purdue Model as your framework
  2. Identify all assets and their communication requirements
  3. Group assets into zones based on criticality and function
  4. Define conduits between zones and specify allowed protocols
  5. Implement firewalls at zone boundaries
  6. Consider data diodes for critical one-way flows
  7. Deploy intrusion detection at key network segments
  8. Document everything in network diagrams and data flow diagrams

Securing Legacy Networks

  1. Conduct a thorough asset inventory
  2. Map all network connections and data flows
  3. Identify "quick wins" like unnecessary Internet connections
  4. Prioritize segmentation of the most critical assets
  5. Implement monitoring before making changes
  6. Test changes in maintenance windows
  7. Document the current state and target state

Secure Remote Access Architecture

üîê Remote Access Requirements

  • VPN termination in IDMZ - never directly into OT
  • Multi-factor authentication for all remote users
  • Jump servers that log all sessions
  • Time-limited access with automatic expiration
  • Network access control limiting access to specific systems
  • Session recording for forensic purposes
